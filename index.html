What is TC39 responsible for?

  • To maintain and update the standard for the ECMAScript programming language.
  • To identify, develop and maintain standards for libraries that extend the capabilities of ECMAScript.
  • To develop test suites that may be used to verify correct implementation of these standards.
  • To contribute selected standards to ISO/IEC JTC 1.
  • To evaluate and consider proposals for complementary or additional technologies.

TC39 - Who is in it?

Members of ECMA are not individuals, they are organizations and corporations.

As such, the members of the Technical committee 39 are organizations, not individuals, but it is attended by individuals representing those organizations.

TC39 - what do they actually do?

They meet on a regular basis and review proposals that are made for the language. They standardize the language and ensure that it conforms to certain requirements

What does a proposal look like?

It is a long process, and is done in four (five) parts!

Stage 0: Strawman

A lot of writing!

1) You will need to find a TC39 Champion (some member on the committee) to pick up your idea and presents it. You can get into this process in a few ways, IRC is a good place to start. Check out the Github repo for more info! https://www.github.com/tc39/proposals/

2) A problem space outline needs to be written -- what are you trying to solve?

3) What kind of algorithms, abstractions, and semantics will be involved?

4) What kind of implementation issues do you forsee?

Stage 1: Proposal

1) you will need a demo. This can take the form of a polyfill or similar

2) you will need to write a spec. A spec defines the feature to be implemented in abstract terms. There are several spec languages, and ECMA Script uses one of them

Expect a LOT of changes and critiques

Stage 2: Draft

Your Initial spec text should cover the following:

all major semantics, syntax and API are covered, but TODOs, placeholders and editorial issues are expected

Technical Committee 39 of ECMA International

Things should get more precise and concrete. You are working towards a final draft of the spec.

expect some (but often incremental) changes

Stage 3: Candidate

It is expected at this point that all Semantics, syntax, and API are fully described by this stage. The solution should be considered complete at this point.

You will also need to write test262 acceptance tests, so that those implementing your feature in the future will be able to run them against your tests.

This is the period in which you will be collecting lots of feedback from users

Requires two complete implementations that have passed your acceptance tests.

You will make a PR to tc39/ecma262 with the integrated spec text!

Stage 4: Finished

Wow, that probably took a long time. But the feature is done!

Now, it is up to browsers and engine writers to implement your feature. It may still be some time before some users use your new feature!
